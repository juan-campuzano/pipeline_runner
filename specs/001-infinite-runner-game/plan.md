## Summary

Build a mobile infinite runner game with Flutter Flame where a "commit" character automatically runs and must be tapped to jump (single + double jump) over procedurally generated obstacles (Bug, Broken Line, Server Down). The game uses Flame's `ParallaxComponent` for infinite scrolling background, `CollisionCallbacks` for game-over detection, and `shared_preferences` for local high score persistence. Sprites sourced from kenney.nl (CC0).

## Implementation Phases

### Phase A — Project Skeleton & Assets (Day 1)

**Goal**: Running project with parallax background visible.

1. `flutter create infinite_runner`
2. Add `flame` and `shared_preferences` to `pubspec.yaml`
3. Download kenney.nl assets → `assets/images/`
4. Declare assets in `pubspec.yaml`
5. Implement `RunnerGame` extending `FlameGame`
   - Set `camera.viewport = FixedResolutionViewport(Vector2(800, 300))` — resolución lógica fija landscape
   - Flame escala y centra el canvas automáticamente en web y móvil (letterboxing)
6. En `main.dart`, antes de `runApp`, llamar `SystemChrome.setPreferredOrientations([DeviceOrientation.landscapeLeft, DeviceOrientation.landscapeRight])` para bloquear landscape en móvil
7. Implement `ParallaxBackground` using `ParallaxComponent` con 3 layers
8. Display parallax background scrolling — smoke test que Flame y el viewport están bien configurados

> **Nota viewport**: Todas las coordenadas del juego (suelo en Y=250, spawner en X=820, etc.) deben referenciarse a la resolución lógica 800×300, nunca al tamaño físico del dispositivo. Esto garantiza comportamiento idéntico en web y móvil.

**Done when**: `flutter run` shows a scrolling background.

---

### Phase B — Player Character (Day 1–2)

**Goal**: Character visible on screen with single and double jump.

1. Implement `PlayerCharacter` as `SpriteAnimationComponent`
   - Load run animation from sprite sheet
   - Load jump frame
2. Ground detection: reset `jumpCount` when `position.y` reaches ground Y
3. Tap handling: `TapCallbacks` mixin on `RunnerGame`; route tap to `player.jump()`
4. Jump logic: apply upward velocity; enforce `jumpCount <= 2`
5. Gravity: apply downward acceleration in `update()`

**Done when**: Character runs, jumps on tap, double-jumps on second tap, lands correctly.

---

### Phase C — Obstacles & Collision (Day 2)

**Goal**: Obstacles spawn and kill the player on contact.

1. Implement `Obstacle` component with `RectangleHitbox`
   - Three types: bug, brokenLine, serverDown
   - Scrolls left at `worldSpeed`; removes itself when off-screen
2. Implement `ObstacleSpawner` with Flame `Timer`
3. Implement `DifficultyManager`
   - Linearly increase `worldSpeed` and decrease `spawnInterval` over time
4. Add `RectangleHitbox` to player + implement `onCollisionStart` → trigger game over
5. Wire `CollisionCallbacks` on `RunnerGame`

**Done when**: Obstacles appear, scroll, and end the game on contact.

---

### Phase D — Score, HUD & Screens (Day 2–3)

**Goal**: Complete game loop with menus and score.

1. Implement `ScoreDisplay` as `TextComponent` (top-right HUD)
   - Increment by elapsed time × multiplier each tick
2. On game over:
   - Pause game loop
   - Load high score from `shared_preferences`
   - Compare and save if new record
   - Show `GameOverScreen` Flutter widget (overlaid via `GameWidget.overlays`)
3. Implement `MainMenuScreen` Flutter widget
4. Implement `GameOverScreen` Flutter widget with score, high score, and "Retry" button
5. "Retry" resets game state and restarts the game loop

**Done when**: Full game loop works: menu → play → game over → retry.

---

### Phase E — Polish & Presentation Prep (Day 3–4)

**Goal**: Demo-ready, stable, visually coherent.

1. Tune difficulty curve (start comfortable, ramp by ~60s)
2. Sprite alignment and hitbox sizing (avoid unfair collisions)
3. Test on target presentation device (Android or browser)
4. Add obstacle type labels or visual hints if time permits
5. Web build smoke test: `flutter build web && flutter run -d chrome`

**Done when**: All 7 presentation checklist items in `quickstart.md` pass.
